# 安全机制效能评估报告

## 1. 评估目标

量化本系统安全机制引入的性能开销与防御有效性，回答两个核心问题：
1. **成本问题**：添加安全机制后，系统运行增加了多少延迟和经济成本？
2. **有效性问题**：安全机制在多少攻击场景下成功拦截了恶意行为？

---

## 2. 延迟开销分析

### 2.1 认证三阶段延迟构成

基于 2v2×3 轮 fullflow 实测数据（6 组完整审计）：

| 阶段 | 安全功能 | 均值(s) | P50(s) | P95(s) | 最大值(s) | 占比 |
|------|----------|---------|--------|--------|-----------|------|
| **Auth (T4)** | DID 解析 + VP 验签 + nonce 防重放 | 3.013 | 0.760 | 7.833 | 8.049 | 21.3% |
| **Probe (T8)** | 就绪性探测 + 工具链验证 + 负载评估 | 10.407 | 11.290 | 11.679 | 11.730 | 73.6% |
| **Context (T12)** | 上下文哈希一致性校验 | 0.728 | 0.710 | 0.902 | 0.925 | 5.1% |
| **全流程** | 三阶段总计 | 14.149 | 12.696 | 20.090 | 20.341 | 100% |

### 2.2 各阶段开销构成分析

```
Auth (21.3%)  ████████░░░░░░░░░░░░  主要开销：链上 DID 文档解析 (~2s, 取决于 RPC 响应速度)
Probe (73.6%) ████████████████████  主要开销：LLM 推理 (~8s) + 工具调用执行 (~2s)
Context (5.1%) █░░░░░░░░░░░░░░░░░░  主要开销：本地哈希计算 (毫秒级)
```

**关键发现**：
- **身份认证 (Auth)** 的安全成本为 ~3s，其中 DID 链上解析占 60-70%。首轮较慢（需从链上首次获取 DID 文档），后续轮次通过缓存降至 ~0.7s
- **就绪性探测 (Probe)** 占总延迟 73.6%，但其主要开销是 LLM 推理，属于**功能成本而非安全成本**
- **上下文校验 (Context)** 的纯安全开销仅 ~0.7s（SHA-256 哈希计算），效率极高

### 2.3 安全机制的净开销

剥离功能性开销后，纯安全机制增加的延迟：

| 安全机制 | 净延迟开销 | 说明 |
|----------|------------|------|
| ECDSA 签名验证 | ~0.1s | VP 签名验签（纯计算） |
| DID 链上解析 | ~2s（首次）/ ~0.1s（缓存后） | 通过 RPC 查询 ERC-1056 |
| Nonce 防重放 | <0.01s | UUID 生成与比对 |
| 上下文哈希校验 | ~0.7s | SHA-256 计算 + 网络往返 |
| CID 内容校验 | ~0.001s | 本地 SHA-256 重算 |
| **合计** | **~2.8s**（首次）/ **~0.9s**（缓存后） | 占全流程 6-20% |

> **结论**：安全机制的净延迟开销为 0.9~2.8 秒，在 14 秒的全流程中占比 6~20%。LLM 推理（~10s）才是主要瓶颈，安全机制的延迟开销处于可接受范围。

### 3.3 大规模运行经济测算（线性外推）

## 5. 互操作性评估

### 5.1 跨角色交互验证

fullflow 测试中验证了以下跨角色交互路径：

```
Verifier C → Holder A (agent_c_op → agent_a_op) : 3轮 ✅
Verifier D → Holder B (agent_d_op → agent_b_op) : 3轮 ✅
```

两条交互路径使用不同的密钥对和 DID，证明系统在标准 DID/VC 协议下支持**任意合法 Agent 间**的互认证。

### 5.2 协议标准合规性

| 标准 | 实现状态 | 说明 |
|------|----------|------|
| W3C DID Core | ✅ | did:ethr:sepolia:0x... 格式 |

| ERC-1056 | ✅ | 链上 DID 注册表 |
| EIP-712 | ✅ | 结构化签名 |

## 6. 效能评估总结

| 维度 | 指标 | 值 | 评价 |
|------|------|----|------|
| **延迟开销** | 安全机制净延迟 | 0.9~2.8s | ✅ 可接受（占全流程 6~20%） |
| **经济成本** | 单次治理成本(Arbitrum) | ~0.29 CNY | ✅ 极低 |
| **攻击拦截** | 负例拦截率 | 100% (4/4) | ✅ 完全拦截 |
| **治理有效性** | 治理流程通过率 | 100% (3/3) | ✅ 闭环验证 |
| **正例可用性** | 正向通过率 | 100% (6/6) | ✅ 安全机制不影响正常交互 |
| **互操作性** | 跨角色互认证 | 2 条路径通过 | ✅ 标准协议互操作 |

> **核心结论**：本系统的安全机制以约 6~20% 的延迟开销和极低的经济成本（<0.3 CNY/治理周期），实现了对 4 类运行时攻击的 100% 拦截，同时不影响正常交互的可用性。安全机制的性能开销主要来自链上 DID 解析，可通过缓存优化至 ~0.9 秒。系统的首要性能瓶颈是 LLM 推理（~10 秒），而非安全机制本身。
