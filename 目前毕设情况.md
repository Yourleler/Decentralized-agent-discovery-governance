# 当前阶段工作总结

## 结论

在学姐原有 `AgentDID / BlockA2A` 身份认证框架的基础上，我们进一步把系统从“只解决认证问题”扩展成了“发现 + 认证 + 治理”的完整闭环架构。  
也就是说，原来主要回答的是“已经找到对方之后，怎么验证它是不是真的”；现在进一步回答了“怎么找到合适的 Agent、怎么筛掉不可靠的 Agent、出了问题之后怎么留痕和治理”。

---

## 1. 学姐原有工作的基础

学姐原有工作已经完成了一个比较完整的去中心化 Agent 身份认证原型，核心包括：

- 基于 DID/VC 的去中心化身份体系
- Holder / Verifier / Issuer 三类运行时
- Verifiable Credential 的申请、签发、出示与验签流程
- Probe 与 Context Check 的运行时状态验证
- 2v2 的认证与审计演示流程

这部分主要解决的是：

- Agent 在去中心化环境里如何拥有可验证身份
- 两个 Agent 已经连接上之后，如何确认对方身份真实
- 如何进一步检查对方当前是否在线、是否具备声明能力、上下文是否一致

它的边界也很明确：

- 默认前提是 Verifier 已经知道 Holder 的 DID 或地址
- 还没有系统性解决“怎么发现 Agent”
- 还没有形成“异常留痕-证据上链-治理处罚-信誉恢复”的完整治理闭环

---

## 2. 我们在此基础上的设计目标

我们这一阶段的目标，是把原有认证框架扩展成一套面向去中心化 Agent 网络的发现与治理架构，让系统同时具备以下能力：

- 能按能力需求去发现 Agent，而不是必须事先知道对方是谁
- 能在发现阶段就先做初步筛选，而不是找到之后再盲目连接
- 能把交互中的异常、欺诈、超时、上下文不一致等行为沉淀为证据
- 能把证据进一步接入链上治理，实现举报、罚没、恢复等动作
- 能在不把高频调用全部上链的前提下，兼顾可追责性与系统效率

一句话概括，就是把系统从“认证原型”推进成“可发现、可验证、可审计、可治理”的混合架构。

---

## 3. 已完成的架构分层



### 3.1 信任与治理层

我们用 **Solidity + Sepolia** 实现了链上注册与治理合约 `AgentRegistry_v1`，把 Agent 的 `DID`、`metadata CID`、质押金额、累计惩罚和治理状态放到链上管理。同时在设计上区分了 **Admin** 和 **OP** 两类角色，前者承担经济责任和治理追责，后者负责日常在线交互与签名。  
这一层完成的目标，是把 Agent 的身份锚点、责任主体和治理状态固定在链上，保证系统具备 **可注册、可追责、可处罚、可恢复** 的基础能力。

### 3.2 评分与质押机制

我们在链上引入了 **质押 + 动态评分** 机制：Agent 注册时需要质押，系统根据质押金额计算初始信誉分 `S_init`；如果后续发生违规，则累计惩罚 `P_total`，并记录最近一次违规时间 `T_last`。在此基础上，链上得到全局信誉分 `S_global`，用于反映 Agent 的长期责任和治理记录。  
同时，我们在 Sidecar 中引入了链下本地评分机制，用 `alpha / beta` 记录正向和负向交互证据，再与链上分融合成最终评分 `S_final`，用于发现阶段的排序。  
这一层完成的目标，是把“经济约束”“历史行为”和“近期表现”统一起来，避免系统只看语义匹配、不看可靠性，也避免恶意节点几乎零成本地反复进入网络。

### 3.3 数据与证据层

我们用 **IPFS / Pinata** 来存储链下数据，包括服务 `metadata`、完整 `VC` 以及违规证据 `PoM`。其中 metadata 用于发现阶段的轻量描述，完整 VC 在认证阶段按需拉取，违规证据在治理阶段按需验证。  
这一层完成的目标，是把大对象和证据材料从链上剥离出来，在保证内容可验证的前提下，降低链上存储成本，同时支持后续的认证核验和治理留痕。

### 3.4 发现与执行层

我们用 **The Graph + Python Sidecar + ChromaDB + SQLite** 构建了本地发现系统。The Graph 负责把链上事件转成可查询的索引视图，Sidecar 负责增量同步链上状态、拉取 metadata、做哈希校验、生成向量索引，并结合语义检索、信誉评分和运行时探测返回候选 Agent。  
这一层完成的目标，是让系统从“知道对方是谁才能验证”变成“可以先按需求发现，再筛选出合适且当前可用的 Agent”，真正补上了原有框架里缺失的 Discovery 环节。

### 3.5 认证与治理闭环

在学姐原有 `Holder / Verifier / Issuer` 认证流程基础上，我们继续复用了 **Auth / Probe / Context Check** 这套机制，并把异常交互进一步接入治理流程：正常交互会更新本地正向证据，异常交互会沉淀为证据并触发举报、罚没、恢复等动作。  
这一层完成的目标，是把系统从单纯的“认证框架”推进成“发现 - 认证 - 审计 - 治理”的完整闭环，使 Agent 网络不仅能连接起来，还能长期稳定运行。


---

## 4. 这一阶段形成的完整闭环

一个完整流程：

1. Agent 注册 DID，上传 metadata，完成链上登记与质押  
2. Subgraph 监听链上事件，把注册和治理状态转成可查询视图  
3. Sidecar 增量同步状态，拉取 metadata，建立本地向量索引  
4. 用户按需求发起语义检索，系统返回候选 Agent  
5. 候选 Agent 进入认证、Probe、上下文检查等验证流程  
6. 若交互正常，则更新本地正向证据与评分  
7. 若交互异常，则沉淀证据，进入举报、处罚、恢复等治理流程  

所以现在系统回答的是一条完整链路：

- 怎么找
- 怎么验
- 怎么筛
- 怎么记
- 怎么罚
- 怎么恢复

## 5. 测试与验证做了什么

这一阶段除了设计和实现，也补了比较完整的测试与验证链路，主要包括：

- **Sidecar 契约测试与行为回归测试**
  - 同步、迁移、幂等、向量同步、本地调分
  - 检索排序、可用性过滤、运行时探测
  - 向量配置与评分相关契约骨架

- **Subgraph 单元测试**
  - 对事件映射和实体落库逻辑进行测试
  - 验证链上事件能否正确转换为查询视图中的实体状态

- **全流程闭环测试 `fullflow_tests`**
  - 覆盖 `Provision -> Discovery -> Verification -> Governance -> Reporting`
  - 支持标准 `2v2`、默认 `3` 轮的完整流程
  - 验证发现链路是否能命中本次注册账户
  - 验证 `Auth / Probe / Context` 三段认证审计流程
  - 验证异常情况下证据是否生成、能否上链举报、能否触发本地治理闭环

- **负例测试**
  - 伪造签名调用 `/auth`，预期被拒绝
  - 中途重置 Holder memory，触发 `Context mismatch`
  - 基于失败证据触发 `reportMisbehavior`
  - 在本地治理脚本中验证 `report -> slash -> restore` 的状态转移

- **结果报表**
  - 输出发现命中、链上交易成本、治理执行结果
  - 输出阶段延迟、通过率、原始指标和摘要报表
  - 为后续论文整理和答辩展示提供数据支撑

## 6. 当前阶段的核心成果

当前这部分工作的核心成果，可以概括为三点：

- 在学姐原有身份认证框架的基础上，补上了“发现”这一关键缺口
- 在认证之后补上了“证据驱动治理”的后半段闭环
- 用分层设计把“链上可信、链下高效、本地可检索、异常可追责”几件事统一了起来
