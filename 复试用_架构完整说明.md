# 复试用架构完整说明

## 1. 项目一句话概括

这个系统做的事情，不是单纯“让 Agent 能聊天”，而是要让 **去中心化环境中的 Agent 能被发现、能被认证、能被审计、出问题还能治理**。

它的核心思路是：

- **链上负责信任锚点和治理锚点**
- **链下负责高频发现、检索、交互和证据整理**
- **DID/VC 负责“你是谁、你宣称了什么”**
- **Sidecar 负责“我怎么找到你、怎么排序你、怎么在本地积累对你的经验”**

这是一套典型的 **混合架构**，而不是把所有东西都硬塞到链上。

---

## 2. 为什么整个系统要这样分层

### 2.1 我们要解决的真实问题

如果只有普通的 Agent 调用接口，会遇到几个问题：

1. 我怎么知道对方是不是它声称的那个 Agent？
2. 我怎么知道它过去是不是作恶过？
3. 全网 Agent 这么多，我怎么按能力找到合适的？
4. 找到了之后，怎么判断它不仅“有身份”，而且“真的能干活”？
5. 如果它在交互时异常、欺诈、超时，怎么留痕、怎么申诉、怎么处罚？

所以这个项目不是只做“身份认证”，而是要把下面几件事连起来：

- 身份
- 发现
- 交互
- 留痕
- 治理

### 2.2 为什么不能全放链上

因为链上适合低频、强一致、可追责的状态，不适合高频交互。

如果把每次 Agent 调用、每次请求响应、每次上下文同步都写链：

- 成本高
- 吞吐低
- 延迟大
- 隐私差
- 完全不适合高频 A2A 场景

所以我们把系统拆成三层：

1. **区块链层**
   负责注册、质押、罚没、申诉、恢复，是唯一事实锚点。
2. **数据层**
   负责 metadata、VC、证据材料等大对象存储。
3. **发现与执行层**
   负责检索、评分、运行时探测、本地留痕、交互编排。

这就是这个架构最核心的设计逻辑。

---

## 3. 架构从头到尾由哪几块组成

整个系统可以分成 10 个核心组成块：

1. **链上注册与治理合约**
2. **Subgraph 索引层**
3. **IPFS / Pinata 数据层**
4. **DID 身份解析与验签层**
5. **VC / VP 凭证体系**
6. **Issuer 发证服务**
7. **Holder Agent 运行时**
8. **Verifier Agent 运行时**
9. **Sidecar 本地发现与评分系统**
10. **Fullflow 全流程实验与验证框架**

下面按这 10 块详细说明。

## 4. 链上注册与治理合约

### 4.1 这块用的是什么

这部分使用的是：

- **Solidity**
- **OpenZeppelin AccessControl**
- **OpenZeppelin ReentrancyGuard**
- **Solady FixedPointMathLib**
- 部署网络是 **Sepolia**

### 4.2 为什么用这些

#### Solidity

因为链上核心状态必须由 EVM 合约承载：

- 注册状态
- 质押
- 罚没
- 恢复
- 申诉入口
- 事件日志

这是整个去中心化治理的基础。

#### OpenZeppelin AccessControl

它解决的是“谁有治理权限”的问题。

我们系统里有治理者角色，所以需要角色权限管理。

为什么不用自己手写权限判断？

- 容易出错
- 容易漏权限边界
- OpenZeppelin 是成熟方案，语义清楚，复用价值高

#### ReentrancyGuard

因为合约里有资金收发，比如：

- 注册时质押
- 退出时退还质押
- slash 时转出罚没资金
- withdrawStake 时转回 ETH

这类地方天然要防重入。

#### Solady FixedPointMathLib

因为初始信誉分 `S_init` 不是线性函数，而是对数函数。

合约里要算：

`S_init = SCORE_BASE + FACTOR * ln(1 + STAKE_MULTIPLIER * stake)`

链上没有好用的原生浮点数学，所以要用高精度数学库。

为什么不用简单线性质押换分？

- 线性模型会让大额质押者无限放大优势
- 对数函数有边际收益递减，更符合“有经济责任但不能纯靠钱碾压”的设计目标

### 4.3 合约里存了什么

合约的核心状态是一个 `Agent` 结构体，里面有：

- `did`
- `metadataCid`
- `initScore`
- `accumulatedPenalty`
- `lastMisconductTimestamp`
- `stakeAmount`
- `isSlashed`
- `isRegistered`
- `isfrozen`
- `admin`

这个设计非常关键，因为它把一个 Agent 在治理层面的核心状态都放齐了。

### 4.4 这块最核心的设计思想

在参数层面，合约里几个最关键的常量是：

- `SCORE_BASE = 60`
- `SCORE_MAX = 100`
- `SCORE_MIN = 40`
- `SCORE_RECOVERY_RATE = 2`
- `FACTOR = 5.5`
- `STAKE_MULTIPLIER = 150`

可以把它们理解成：

- 新注册 Agent 的信誉下限从 60 起步
- 质押金额越高，初始分越高，但采用对数增长，不是线性暴涨
- 最高不超过 100
- 如果全局分跌到 40 或以下，就进入非常危险的治理区间
- 每天最多恢复 2 分，但恢复也不会超过该 Agent 自己的初始上限

#### 1. Admin 和 OP 分离

系统强调：

- **Admin** 是链上注册主体，也是经济责任主体
- **OP** 是日常在线交互使用的操作密钥

也就是说：

- 链上罚没罚的是 Admin
- 在线服务跑的是 OP

这样做的原因是：

- 避免把高价值私钥暴露在在线进程里
- 让“交互密钥”和“经济责任密钥”分离
- 出事时仍然能追责到真正责任主体

#### 2. 合约只存“轻状态”

链上不存完整 metadata，不存完整 VC，不存完整交互日志。

只存：

- DID
- Metadata CID
- 分数相关状态
- 质押状态
- 治理状态

这是因为链上适合做锚点，不适合做大对象存储。

### 4.5 合约暴露了哪些核心函数

#### 1. `registerAgent`

作用：

- 注册 DID
- 绑定 metadata CID
- 接收初始质押
- 计算初始信誉分

它是 Agent 上链进入系统的入口。

#### 2. `unregisterAgent`

作用：

- 注销 Agent
- 返还剩余质押

但这里不是简单删除，而是保留历史痕迹，不允许 DID 随便洗白重来。

#### 3. `updateServiceMetadata`

作用：

- 更新 metadata 对应的 CID

这使得 Agent 的描述信息可以演进，但链上只存 CID，不存大对象。

#### 4. `depositStake`

作用：

- 增加质押
- 重算 `initScore`

#### 5. `withdrawStake`

作用：

- 减少质押
- 重算 `initScore`
- 但要求减持后全局分不能掉到最低阈值以下

这保证 Agent 不能在享受高信誉的同时把质押偷偷抽走。

#### 6. `reportMisbehavior`

作用：

- 举报某个 Agent 的违规行为
- 链上只记录证据 CID，不做证据重验证

也就是说，合约只做证据锚定入口，不做复杂仲裁逻辑。

#### 7. `appeal`

作用：

- 被处理的 Agent 可以提交申诉证据 CID

#### 8. `freeze`

作用：

- 治理者冻结一个 Agent
- 冻结后它不能通过注销或提款规避处理

#### 9. `unfreeze`

作用：

- 解冻

#### 10. `slash`

作用：

- 增加罚分
- 更新违规时间
- 扣减质押
- 如果分数太低，标记为 slashed

这一步是治理惩罚的核心。

#### 11. `restore`

作用：

- 治理者在申诉成立时恢复 Agent 状态
- 重置部分罚分
- 重置违规时间
- 解除 slashed

注意它不会凭空补钱，资金部分需要 Agent 自己重新补质押。

#### 12. `getGlobalScore`

作用：

- 按链上规则计算当前全局信誉分

公式语义是：

`S_global = min(S_init, S_init - P_total + 恢复项)`

恢复项按“距离上次违规过去多少天”计算。

### 4.6 合约会触发哪些事件

核心事件包括：

- `AgentRegistered`
- `AgentUnregistered`
- `ServiceUpdated`
- `MisbehaviorReported`
- `AgentAppealed`
- `AgentSlashed`
- `AgentRestored`
- `StakeUpdated`
- `TreasuryUpdated`

此外因为用了 AccessControl，还会有：

- `RoleGranted`
- `RoleRevoked`
- `RoleAdminChanged`

### 4.7 这块是自己写的还是库做的

- **自己写的**
  - Agent 数据结构
  - 注册/质押/治理逻辑
  - 信誉分设计
  - 事件语义
- **库提供的**
  - AccessControl 权限框架
  - ReentrancyGuard 防重入
  - 固定点数学工具

## 5. Subgraph 索引层

### 5.1 这块用的是什么

- **The Graph / Subgraph**
- GraphQL 查询接口
- Mapping 逻辑由 AssemblyScript 编写

### 5.2 为什么要加 Subgraph，而不是 Sidecar 直接扫链

因为直接扫链有几个问题：

- 要自己处理事件分页
- 要自己处理历史区块回放
- 要处理重组、断点、水位线等复杂问题
- 每个 Sidecar 都扫链会重复浪费资源

Subgraph 的作用就是把链上事件变成一个已经整理好的可查询视图。

所以它不是替代区块链，而是做：

- 事件收集
- 状态投影
- GraphQL 查询服务

### 5.3 Subgraph 做了哪些事

它把合约事件分成两类实体：

#### 1. immutable 事件实体

例如：

- 注册事件
- 举报事件
- 申诉事件
- 罚没事件
- 恢复事件

这些是原始日志快照，只增不改。

#### 2. mutable 聚合实体

最重要的是 `Agent` 聚合实体，它维护一个 Agent 的当前状态：

- did
- cid
- initScore
- stakeAmount
- accumulatedPenalty
- lastMisconductTimestamp
- slashed
- isRegistered
- lastUpdatedBlock

这相当于给 Sidecar 提供了一张“当前可发现 Agent 状态表”。

### 5.4 为什么这样设计

因为系统既需要：

- 原始可追溯事件
- 又需要可直接查询的当前状态

如果只有原始事件，Sidecar 每次都要自己回放历史。

如果只有当前状态，又丢掉了治理追责的时间线。

所以两个都要。

### 5.5 这块是自己写的还是现成的

- **自己写的**
  - 事件到实体的映射逻辑
  - GraphQL Schema 设计
  - 哪些事件改聚合态、哪些只记日志的规则
- **平台提供的**
  - The Graph 节点运行能力
  - GraphQL 查询接口能力

## 6. 数据层：IPFS / Pinata / 本地缓存

### 6.1 这块用的是什么

- **IPFS**
- **Pinata REST API**
- **httpx**
- **tenacity**
- 本地 `.ipfs_cache`

### 6.2 为什么用 IPFS

因为系统有很多不适合上链的大对象：

- metadata
- VC
- 申诉证据
- 治理证据

这些对象的特点是：

- 容量大
- 更新频率不一致
- 需要内容寻址

IPFS 非常适合做这种“链下内容存储 + 链上 CID 锚定”。

### 6.3 为什么还要 Pinata

因为原生 IPFS 有“没人 pin 可能找不到”的问题。

Pinata 的作用是：

- 帮我们 pin 内容
- 提供稳定上传接口
- 提供专属网关

所以它解决的是“IPFS 可用性”问题，不是替代 IPFS。

### 6.4 为什么不用数据库代替 IPFS

因为普通数据库是中心化的，不具备：

- 内容寻址
- CID 锚定
- 多方可验证的对象标识

我们的目标不是只把文件存起来，而是要让这个文件可以被链上状态引用和校验。

### 6.5 这层具体怎么工作

#### 上传时

系统把 JSON 或文件上传到 Pinata，对应得到 CID。

典型用途：

- metadata 上传后，CID 写到链上
- 证据文件上传后，CID 用于举报或申诉

#### 下载时

系统先查本地缓存。

如果缓存没有，就发起网关下载。

下载不是只打一个网关，而是做 **网关竞速**：

- 专属网关优先
- 再加公共网关备份
- 谁先返回就用谁

这样做是为了降低单网关不稳定带来的问题。

#### 校验时

下载后会做 SHA256 校验，并把原始内容放进本地缓存。

### 6.6 这层是自己写的还是库做的

- **自己写的**
  - Pinata REST 封装
  - 网关竞速策略
  - 本地缓存逻辑
  - 下载后校验流程
- **用到的库**
  - `httpx` 做 HTTP 请求
  - `tenacity` 做重试

## 7. DID 身份解析与验签层

### 7.1 这块用的是什么

- **did:ethr**
- **Web3.py**
- **eth_account**
- Node 侧的 DID 解析脚本

### 7.2 为什么用 DID，而不是传统账号体系

因为传统中心化账号体系的问题是：

- 依赖中心化注册方
- 不适合 Agent 自主身份
- 不适合链上责任绑定

而 DID 的好处是：

- 身份可独立存在
- 可解析控制权
- 可挂接 delegate key
- 可和链上地址自然绑定

### 7.3 为什么选 did:ethr

因为你们整个系统本来就建立在 EVM 链上：

- 注册合约在 EVM
- 责任主体是以太坊地址
- 签名也是以太坊签名

所以 did:ethr 是最自然的选择。

如果改成别的 DID 方法，会增加系统复杂度，不利于本项目的“工程可落地性”。

### 7.4 验签层怎么工作

#### 第一步：解析 DID

系统会调用 Node 脚本去解析 DID 文档。

这里不是纯 Python 解析，而是借助 Node 生态。

原因是以太坊 DID 解析的成熟工具链更多在 JS 侧。

#### 第二步：恢复签名地址

收到签名后，用以太坊签名恢复机制恢复出签名地址。

#### 第三步：检查是否被 DID 文档授权

系统会遍历 DID 文档中的 `verificationMethod`：

- 看是不是 owner
- 看是不是 delegate

如果恢复出来的地址在授权列表里，就通过。

#### 第四步：必要时清缓存重试

如果缓存的 DID 文档验证失败，会清缓存再查一次链上，避免因为陈旧缓存误判。

### 7.5 为什么不只信地址，不解析 DID 文档

因为系统不是只做“地址签名”，而是做 **DID 控制权验证**。

只有地址签名，无法表达：

- delegate key 是否被授权
- admin / op 关系
- DID 语义身份

所以必须解析 DID 文档。

### 7.6 这层是自己写的还是库做的

- **自己写的**
  - DID 验签流程编排
  - 缓存 + 重试 + 授权检查逻辑
- **用到的库**
  - Web3.py
  - eth_account
  - did 解析依赖主要通过 Node 脚本侧完成

## 8. VC / VP 凭证体系

### 8.1 这块用了什么

- W3C 风格的 **VC**
- W3C 风格的 **VP**
- 自定义 JSON Schema
- ECDSA secp256k1 签名

### 8.2 为什么需要 VC

因为 DID 只回答“你是谁”，但不回答：

- 你是什么能力
- 你用什么模型
- 你是否合规
- 你是否通过某些审计

VC 负责承载“声明”。

### 8.3 为什么需要 VP

因为交互时不能直接说“我有证书”，而是要把证书以一个带挑战值的 presentation 形式出示给验证方。

VP 的意义是：

- 把多张 VC 组合起来
- 由 Holder 再签一次
- 与当前 challenge 绑定

这可以防止简单重放。

### 8.4 系统里定义了哪些 VC 类型

目前你们定义了五类主 VC：

1. 身份溯源
2. 核心模型
3. 能力基准
4. 工具链审计
5. 安全合规

这使得系统不是只验证单一身份，而是从多个维度验证 Agent。

### 8.5 VP 验证流程怎么走

验证 VP 时，系统会做：

1. 检查 challenge 是否等于预期 nonce
2. 去掉 proof 后序列化主体
3. 验 Holder 的签名
4. 逐张验 VC

逐张验 VC 又包括：

1. 检查 `credentialSubject.id`
2. 检查是否过期
3. 检查 issuer 是否受信任
4. 检查 issuer 对该 VC 的签名

### 8.6 为什么不做特别复杂的 VC 机制

当前系统是 **工程化 MVP**，不是完整 SSI 平台。

所以选择了：

- 足够清晰的最小 VC 结构
- 可以直接被 Python 运行时处理
- 可以和现有 DID 签名逻辑衔接

而没有引入特别复杂的：

- 零知识证明
- 选择性披露
- 状态列表链上联动
- 多重委托链

这不是做不到，而是当前研究重点不在那。

### 8.7 信任寄存怎么处理

系统现在新增了一个最小 **Issuer Trust Registry**。

它的默认策略是：

- 没有单独配置时，默认全部允许
- 如果以后要加强，可切换到白名单模式

这意味着你们已经给“谁有资格发证”预留了接口，但当前运行策略仍然是简单模式。

### 8.8 这块是自己写的还是库做的

- **自己写的**
  - VC / VP 结构设计
  - 验证流程
  - issuer trust registry
- **用到的库**
  - Web3.py / eth_account 负责签名与验签

## 9. Issuer 发证服务

### 9.1 这块是什么

Issuer 是一个独立服务，扮演“发证机构”角色。

### 9.2 用了什么

- Flask
- Web3.py / eth_account
- DIDValidator
- 本地 VC 模板集合

### 9.3 为什么要独立成服务

因为发证逻辑和验证逻辑、运行时逻辑不是一回事。

独立成服务的好处是：

- 角色清晰
- 方便模拟真实世界的第三方发证机构
- 便于未来替换成 DAO、组织、审计机构

### 9.4 它具体怎么工作

当 Holder 向 Issuer 发起申请时：

1. Holder 提交申请 payload
2. 带上自己的 DID 和签名
3. Issuer 验签，确认申请者真的是该 DID 的控制者
4. Issuer 从模板集合里生成多张 VC
5. 自动补 issuer、有效期等字段
6. 对每一张 VC 进行签名
7. 返回 VC 列表

### 9.5 为什么使用模板签发

因为系统要验证的是“多维属性”，而不是一张统一证。

模板化的好处是：

- 结构清楚
- 易于扩展
- 易于控制不同类型 VC 的字段

### 9.6 这块是自己写的还是库做的

- **自己写的**
  - 发证接口
  - 模板加载与替换逻辑
  - Proof 包装逻辑
- **用到的库**
  - Flask
  - Web3.py / eth_account

## 10. Holder Agent 运行时

### 10.1 这块是什么

Holder 是“被验证、被探测、被检查上下文”的一方。

它不是单纯的 HTTP 服务，而是“Agent Runtime + 身份钱包 + 凭证钱包 + 审计接口”的组合。

### 10.2 用了什么

- Flask
- IdentityWallet
- DIDValidator
- RuntimeStateManager
- LangChain Agent
- DashScope / QwQ 模型
- 两个工具函数：
  - 求 SHA-256
  - 获取当前 UTC 时间

### 10.3 为什么这样设计

Holder 被拆成两层：

#### 1. Runtime

负责：

- 接口接收
- 验签
- 调钱包
- 组织 VP
- 记录交互

#### 2. Brain

由 LLM 驱动，负责：

- 是否同意认证
- 是否同意上下文审计
- 如何完成 probe task

这样分层的原因是：

- 私钥和签名逻辑不能交给 LLM
- LLM 只负责决策与任务处理
- Runtime 负责安全边界

### 10.4 Holder 的主要流程

#### 启动时

Holder 会先自检本地是否已有可用 VC。

如果没有，就向 Issuer 发起 VC 申请。

#### 认证阶段

Verifier 发来认证请求后：

1. Holder 先验对方签名
2. Agent Brain 判断是否同意认证
3. 如果同意，就由钱包生成 VP
4. 把 VP 返回给对方

#### Probe 阶段

Verifier 发来任务后：

1. Holder 验对方签名
2. LLM Agent 结合工具完成任务
3. Runtime 把结果打包并签名
4. 返回执行结果

#### Context 阶段

Verifier 发来上下文哈希检查请求后：

1. Holder 验对方签名
2. 计算自己当前会话视角下的上下文快照哈希
3. LLM 决定是否同意出示
4. Runtime 返回带签名的 context hash

### 10.5 Holder 为什么要做上下文哈希

因为仅仅验证“你这次回答得对”，还不够。

系统还想验证：

- 你现在的内部会话上下文是不是和交互过程一致
- 你是不是偷偷换状态了

所以引入了 **Context Consistency Check**。

### 10.6 Holder 的本地记录怎么做

现在的实现已经统一成：

- 优先写入 SQLite
- 如果 SQLite 出问题，才退回旧 JSON 兼容方式

这一步的意义非常大，因为它把原来零散调试式记录，往可申诉、可导出、可统计的方向推进了。

### 10.7 这块是自己写的还是库做的

- **自己写的**
  - Runtime 接口
  - 认证/探测/上下文检查流程
  - VP 返回逻辑
  - SQLite 留痕接线
- **用到的库**
  - Flask
  - LangChain
  - DashScope/QwQ

## 11. Verifier Agent 运行时

### 11.1 这块是什么

Verifier 是主动发起审计的一方。

它不只是收结果，而是整个审计流程的控制器。

### 11.2 用了什么

- requests
- IdentityWallet
- DIDValidator
- RuntimeStateManager
- LangChain Controller
- LangChain Judge
- DashScope / QwQ 模型

### 11.3 为什么 Verifier 也拆成两类智能能力

Verifier 里实际上有两个 LLM 角色：

#### 1. Controller

作用：

- 按状态机决定下一步做什么
- 例如发起认证、发起 probe、发起 context check、终止流程

#### 2. Judge

作用：

- 在 probe 结果出来后，评估回答内容是否合理

这种拆分的好处是：

- 流程控制和内容审计分离
- 容易调试
- 不会让一个 LLM 同时兼顾太多职责

### 11.4 Verifier 的主要流程

#### 第一步：请求 VC

如果自己缺证，会先向 Issuer 请求 VC。

#### 第二步：发起认证

它构造一个带 nonce 的认证请求，发给 Holder。

收到 VP 后：

- 验证 challenge
- 验 Holder 签名
- 验里面的每一张 VC

#### 第三步：发起 probe

Verifier 不直接手写任务，而是：

- 从模板池和输入池中随机抽取
- 动态生成任务
- 设置 timeout
- 发给 Holder

收到返回后：

- 先检查工具结果是否正确
- 再让 Judge 审查总结内容是否合理

#### 第四步：发起上下文校验

Verifier 请求 Holder 返回 context hash。

然后 Verifier 拿自己本地保存的同一会话交互记录重新算哈希。

如果两边哈希一致，则说明上下文记录一致。

### 11.5 为什么 probe 要用模板，而不是自由提问

因为自由提问太难客观评估。

模板化 probe 的好处是：

- 可预先知道正确答案
- 可测 SLA
- 可做自动审计
- 便于正负例构造

### 11.6 为什么还要 AI Judge

因为有些任务不是纯字符串匹配就能验证。

例如摘要任务里：

- 同样意思可以有不同表述
- 不能只看某个固定文本

所以需要 Judge 做语义层面的合理性判断。

### 11.7 Verifier 的本地记录怎么做

和 Holder 一样，已经切到 SQLite 优先。

Verifier 本地记录的作用有三类：

1. 用于 context hash
2. 用于申诉导出
3. 用于统计目标 Agent 的近 7 天调用次数

### 11.8 这块是自己写的还是库做的

- **自己写的**
  - 审计状态机
  - Probe 任务生成
  - 工具结果验证
  - Context 校验
  - 本地记录接线
- **用到的库**
  - requests
  - LangChain
  - DashScope/QwQ

## 12. Sidecar：本地发现与评分系统

### 12.1 这块是什么

Sidecar 是整个架构里最“工程化”的部分。

它把链上、链下、检索、评分、本地状态管理统一起来。

如果说：

- 合约是系统的法典
- DID/VC 是系统的身份证

那 Sidecar 就是系统真正“每天在干活的大脑”。

### 12.2 Sidecar 由哪几个小模块组成

可以拆成 5 个部分：

1. **SQLite 状态存储**
2. **Subgraph 客户端**
3. **同步编排器**
4. **向量索引层**
5. **发现服务**

### 12.3 SQLite 状态存储

#### 存了什么

最重要的表有：

##### 1. `agent_state`

记录每个 Agent 的本地状态，包括：

- did
- metadata cid
- init_score
- accumulated_penalty
- last_misconduct_timestamp
- alpha / beta
- global_score
- local_score
- confidence_score
- final_score
- metadata_sha256
- vector_text
- runtime_probe_url
- 探测时间和失败次数
- `recent_7d_calls`

##### 2. `sync_state`

记录同步水位线，例如 `last_synced_block`。

##### 3. `interaction_receipt`

这是最近新增的精简交互记录表，记录：

- owner_did
- peer_did
- caller_did
- target_did
- session/task 信息
- request / response JSON
- request / response hash
- stage
- status
- latency
- source
- created_at

#### 为什么要用 SQLite

因为 Sidecar 是本地单节点系统，SQLite 非常适合：

- 零部署
- 本地持久化
- 支持事务
- 结构清楚
- 便于做统计与导出

为什么不用 Redis？

- Redis 偏内存，不适合这里的持久留痕

为什么不用 PostgreSQL？

- 太重，不符合项目最小落地原则

### 12.4 最近 7 天调用次数怎么实现

这个问题现在已经被补上了。

实现方式非常简单：

1. 每插入一条交互记录，都带 `created_at`
2. 以 `target_did` 为维度，统计最近 7 天内有多少条记录
3. 把这个结果回写到 `agent_state.recent_7d_calls`

这种做法的特点是：

- 简单
- 不需要定时任务
- 不需要复杂窗口维护
- 足够支持你们现在的发现排序与复试回答

### 12.5 同步编排器做什么

同步编排器的职责是：

1. 读水位线
2. 拉 Subgraph 增量
3. 转成本地 AgentState
4. 计算评分
5. 按需拉 metadata
6. 同步向量索引
7. 推进水位线

它是 Sidecar 把“链上状态”变成“本地可检索状态”的关键枢纽。

### 12.6 为什么只增量同步，不全量扫

因为发现系统必须常态运行。

如果每次启动都从头扫全量：

- 很慢
- 很浪费
- 全网规模上来后根本不可用

所以必须用：

- Subgraph 聚合视图
- 水位线
- 增量拉取

### 12.7 Sidecar 的评分模型

这部分是复试非常容易被问的重点。

系统评分分三层：

#### 第一层：链上全局分 `S_global`

语义是：

- 初始质押越多，初始分越高
- 违规越多，扣分越多
- 随时间可缓慢恢复

这是“客观治理分”。

这一层和链上参数保持一致：

- 全局恢复速率：`2 分/天`

#### 第二层：链下本地分 `S_local`

它由 `alpha / beta` 两个证据量推导出来。

- `alpha` 表示正向证据
- `beta` 表示负向证据

实现里有几个关键步骤：

##### 1. 时间遗忘

对 `alpha / beta` 做指数衰减，避免特别久远的行为永久影响当前判断。

##### 2. 无违规正向恢复

如果一段时间没有再被链上认定违规，会给 `alpha` 增加恢复量。

##### 3. 计算局部分

先求：

`p = alpha / (alpha + beta)`

再得到：

`S_local = 1 + k * (p - 0.5)`

然后做上下界裁剪。

这一步的本质是：

- 本地经验可以影响排序
- 但不能无限放大

这一层在实现中用了明确参数：

- 时间遗忘系数 `gamma = 0.9999997327`
- 每日正向恢复 `r = 0.2`
- 局部分调节系数 `k = 0.8`
- 局部分裁剪范围 `delta = 0.2`

所以 `S_local` 不是无限波动的，而是一个受控的、温和的、本地经验修正项。

#### 第三层：置信度权重 `w`

如果证据太少，就不能让 `S_local` 的影响太大。

所以又做了一个置信度压缩：

- 证据越多，局部分影响越可信
- 证据越少，局部分影响越保守

这一层也用了明确参数：

- `C = 10`
- `lambda = 0.15`

#### 第四层：最终分 `S_final`

最终公式是：

`S_final = S_global * S_local * w`

这表示：

- 链上客观分是基础
- 链下主观经验可以微调
- 证据多少决定微调力度

### 12.8 为什么这样设计，不直接只看链上分

如果只看链上分，会有几个问题：

- 发现结果太“僵”
- 不能反映本地近期体验
- 无法区分“同样注册但实际可用性不同”的 Agent

如果只看链下分，也不行：

- 容易被局部观察污染
- 缺乏全局治理约束

所以必须做链上和链下混合。

### 12.9 向量检索层做什么

这部分用的是：

- ChromaDB
- sentence-transformers embedding function
- 默认模型是 `BAAI/bge-m3`

#### 为什么用 Chroma

因为项目要做本地语义检索，不只是标签过滤。

Chroma 的优点是：

- 嵌入式部署简单
- 本地持久化方便
- 适合原型系统

为什么不用 Elasticsearch / Milvus / FAISS 单独封装？

- Elasticsearch 偏全文检索，语义向量不是最轻量方案
- Milvus 太重
- FAISS 还需要自己补很多持久化和接口层

#### 为什么用 BGE-M3

从实现选择看，这里主要是为了：

- 中英文兼容较好
- 适合多语语义检索
- 可以直接被现有 embedding 封装调用

这是一个兼顾效果和工程简洁度的选择。

### 12.10 metadata 为什么要单独设计 schema

因为发现阶段不应该一上来就下载完整 VC。

所以 metadata 承担的是“轻量发现描述”的作用。

它包含：

- 基本服务信息
- capabilities
- interactionModes
- endpoint
- vcManifest
- indexHints
- 时间戳

这个 schema 的意义是：

- 给 Sidecar 做向量化
- 给用户做服务概览
- 为后续 lazy fetch VC 预留入口

### 12.11 发现服务怎么排序

发现服务不是直接按向量距离排序，而是：

1. 先向量召回
2. 做距离窗口过滤
3. 计算语义分
4. 计算信誉压缩分
5. 做信誉微调
6. 对前几名做运行时探测
7. 过滤不可用 Agent

它的核心思路是：

- **语义优先**
- **信誉微调**
- **可用性兜底**

实现里排序和过滤还用了明确参数，这一点在复试时很值得强调，因为它说明你们的设计不是拍脑袋：

- 召回放大倍数：`8`
- 最小召回条数：`40`
- 最大语义距离阈值：`0.55`
- 距离窗口：`0.22`
- 语义衰减系数：`3.2`
- 信誉压缩中心：`60`
- 信誉压缩尺度：`14`
- 信誉倍率基线：`0.90`
- 信誉倍率增益：`0.20`
- 只对前 `3` 名候选做在线探测
- 探测结果缓存 TTL：`120 秒`
- 探测超时：`1.5 秒`
- 连续失败阈值：`3`
- 冷却时间：`600 秒`

这说明最终返回结果不是简单的“谁向量最近就返回谁”，而是综合考虑：

- 功能是否匹配
- 历史信誉是否可靠
- 当前服务是否还活着

### 12.12 为什么不是只看 final_score

因为发现本质上先是“功能匹配”，不是“名气排名”。

如果只看 final_score，会导致：

- 一个高分但功能不匹配的 Agent 排到前面

所以必须是：

- 先语义
- 再信誉

这点答辩时很重要。

### 12.13 运行时探测为什么要做

因为一个 Agent 即使：

- 注册了
- 有分
- metadata 也没问题

它也可能：

- 当前服务挂了
- 接口无响应
- 长期超时

所以 discovery 返回前，还要对少量候选做轻探测。

系统记录：

- 最近探测时间
- 最近成功时间
- 连续失败次数

并设置冷却逻辑。

### 12.14 这块是自己写的还是库做的

- **自己写的**
  - 同步编排器
  - 评分公式
  - metadata 提取规则
  - discovery 排序逻辑
  - recent_7d_calls 与 interaction_receipt 设计
- **用到的库**
  - SQLite
  - ChromaDB
  - sentence-transformers
  - requests / httpx

## 13. 运行时本地状态与申诉证据

### 13.1 最近这次新增了什么

最近补的最关键能力有两点：

1. **把交互留痕从零散 JSON 收敛成 SQLite**
2. **加了最小化 issuer trust registry**

### 13.2 为什么要收敛到 SQLite

因为原始 JSON 记录方式的问题是：

- 不方便统计
- 不方便做 7 天计数
- 不方便统一导出申诉证据
- 不方便跨模块复用

SQLite 后，系统可以统一支持：

- 上下文快照哈希
- 交互计数
- 申诉文件导出
- 本地分析

### 13.3 申诉证据怎么准备

现在底层能力已经具备：

1. 从 `interaction_receipt` 提取指定对端的交互记录
2. 组装成标准化 payload
3. 后续可以直接把 payload 上传 IPFS
4. 再把 CID 提交给申诉或举报入口

这就是“链下组织证据，链上只锚定证据”的思路。

### 13.4 为什么不把申诉细节直接全上链

因为：

- 太贵
- 太重
- 还会泄露细节

所以正确做法是：

- 明细链下
- CID 链上

## 14. Fullflow 全流程实验与验证框架

### 14.1 这块是什么

Fullflow 不是业务主链路的一部分，但它很重要，因为它把整套架构串成一个可复现实验闭环。

### 14.2 它包含哪些阶段

#### 1. Provision

准备账户、配置、私钥、测试环境。

#### 2. Discovery

启动 Issuer、Holder、Verifier，执行 2v2 审计流程。

- 原始记录

### 14.3 为什么要专门做 fullflow

有了 fullflow，才能证明：

- 发现真的能命中
- 认证真的能通过
- probe 真的能完成
- context 真的能校验
- 治理真的能提交流程

这对复试答辩非常重要，因为它说明你们不是纸上设计。

## 15. 整个系统从头到尾的大流程

下面用一条完整链路把全系统串起来。

### 阶段 1：注册与上链

1. Agent Admin 准备 DID 和 OP 体系
2. 生成 metadata
3. metadata 上传到 IPFS
4. 得到 metadata CID
5. 调用合约注册
6. 合约记录 DID、CID、质押和初始分
7. 合约触发注册事件

### 阶段 2：索引与同步

1. Subgraph 监听到注册事件
2. 更新 Agent 聚合实体
3. Sidecar 通过 GraphQL 拉增量
4. Sidecar 把结果转成本地状态
5. Sidecar 拉 metadata
6. 做 SHA256 校验
7. 提取向量文本
8. 写入 Chroma
9. 计算本地评分并存入 SQLite

### 阶段 3：发证

1. Holder 启动发现自己没有 VC
2. Holder 向 Issuer 发起申请
3. Issuer 验签
4. Issuer 用模板签发 VC
5. Holder 本地保存 VC

### 阶段 4：发现

1. Agent A 提出能力需求
2. Sidecar 做语义检索
3. 结合 `S_final` 做排序
4. 对前几名做运行时探测
5. 返回当前最合适的 Agent

### 阶段 5：认证

1. Verifier 发起认证请求
2. Holder 验对方签名
3. Holder 决定是否同意认证
4. Holder 返回 VP
5. Verifier 验 VP 与 VC

### 阶段 6：探测

1. Verifier 生成 probe task
2. Holder 执行任务
3. Verifier 校验结果
4. Judge 做内容审计

### 阶段 7：上下文检查

1. Verifier 请求 context hash
2. Holder 计算本地上下文快照哈希
3. Verifier 用自己本地记录算哈希
4. 两边比对是否一致

### 阶段 8：留痕

整个过程中，双方都会把交互写入本地 SQLite。

这些记录用于：

- 上下文校验
- 近 7 天调用统计
- 申诉证据导出

### 阶段 9：治理

如果出现异常：

1. 组织证据
2. 上传到 IPFS
3. 拿到 evidence CID
4. 调用举报或申诉入口
5. 治理者执行 slash / restore / freeze / unfreeze
6. 事件继续被 Subgraph 同步
7. Sidecar 下一轮同步后更新本地状态

这就形成了一个闭环。

## 16. 复试时很可能被问到的“为什么不用别的”

### 16.1 为什么不用中心化注册表

因为项目目标就是去中心化 Agent 发现与治理。

中心化注册表的问题是：

- 单点失效
- 不可验证
- 平台可随意删改

### 16.2 为什么不用链上全量存储

因为太贵、太慢、不适合高频交互。

### 16.3 为什么不用纯链下信誉

因为没有强约束，容易变成谁都能各说各话。

### 16.4 为什么不用纯链上信誉

因为太硬，无法反映近期本地体验，也不适合高频更新。

### 16.5 为什么不用纯关键词搜索

因为 Agent 能力描述是自然语言，语义检索更合理。

### 16.6 为什么不用只验证 DID，不验证 VC

因为 DID 只证明身份，不证明能力、模型、合规、工具链等声明。

### 16.7 为什么不用只验证 VC，不做 probe

因为有证不代表真会干活。

### 16.8 为什么不用只做 probe，不做 context check

因为一次对答正确，不代表内部状态没有问题。

### 16.9 为什么不用直接把 LLM 当系统核心

因为：

- LLM 不适合掌管私钥
- LLM 不适合直接承担可追责状态
- LLM 负责判断和处理任务可以
- 状态、签名、治理必须放在 Runtime 和链上

## 17. 当前系统已经做到的，与还没做到的

### 17.1 已经做到的

- DID 身份验证
- VC / VP 认证
- Issuer / Holder / Verifier 闭环
- Probe 审计
- Context consistency check
- 链上注册、质押、举报、申诉、罚没、恢复
- Subgraph 增量索引
- Sidecar 本地发现、评分、检索
- 本地 SQLite 交互留痕
- 最近 7 天调用次数统计
- 申诉证据导出底层能力

### 17.2 还没有完整做的

- 真正协议级的 A2A 标准兼容
- 全网公开的调用热度榜
- 去中心化统一 issuer trust network
- 更复杂的自动仲裁机制
- 更高级的 VC 状态同步与吊销体系

### 17.3 这是不是缺点

不一定。

更准确地说，这是 **按研究目标和工程成本做的有意识收敛**。

因为本项目的重点是：

- 去中心化发现
- 信任认证
- 证据驱动治理

而不是一步做成完整工业级 SSI 平台。

## 18. 一段适合复试时直接说的总结

如果老师问“你这个系统到底是什么”，可以用下面这段总结：

> 我们做的是一个面向去中心化 Agent 网络的发现与治理系统。它不是单点的聊天机器人平台，而是一套把链上治理、链下检索、DID 身份、VC 凭证、运行时审计和本地信誉评分结合起来的混合架构。链上合约负责注册、质押、罚没和申诉，是系统的信任锚点；Subgraph 负责把事件变成可查询状态；Sidecar 在本地维护 Agent 状态、拉取 metadata、做向量检索和评分排序；Holder、Verifier、Issuer 三个运行时负责发证、认证、探测和上下文一致性校验；如果发生异常，交互记录会在链下整理成证据，再通过 CID 上链进入治理流程。这样做的目的是在不把高频调用全上链的前提下，同时保证系统可发现、可认证、可追责、可恢复。  

---

## 19. 一句话点明你们当前架构的最大价值

这个架构最大的价值，不是“让 Agent 能互相发消息”，而是 **让 Agent 在去中心化环境下的身份、发现、交互、信誉和治理形成闭环**。
